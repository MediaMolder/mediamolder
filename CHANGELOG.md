## [Unreleased]

### Fixed
- **Positional FFmpeg-style filter args (`scale=320:240`) reach libavfilter intact.** `compat/ffcli/ffcli.go.parseFilterExpr` records each `:`-separated token without an embedded `=` as `_pos0`, `_pos1`, ... so the original order can be recovered, but `pipeline/engine.go.buildFilterSpec` previously emitted those keys verbatim, producing `scale=_pos0=320:_pos1=240` which `avfilter_graph_parse_ptr` rejected with `Could not set non-existent option '_pos0'`. The builder now partitions params into positional (`_posN`) and named, sorts positional by index, and emits them as bare in-order values before any named args. Surfaced by the `vf_scale_positional_x264_audio_copy` round-trip case.
- **libx264 (and any encoder) fed by a filter graph no longer blows up the container duration.** When the upstream filter graph's buffersink advertised a finer `time_base` than `1/framerate` (e.g. demuxer TB `1/12288` propagated through `scale`), the encoder was opened with `time_base = 1/framerate` only, so frame PTS produced in the buffersink's units were reinterpreted in the encoder's coarser units — a 24 fps clip ran out at ~512× the real duration (219 frames → ~4650 s instead of ~9.1 s). `av.EncoderOptions` gained a `TimeBase [2]int` field that, when set, overrides the framerate-derived default; `pipeline/handlers.go.createEncoder` now populates it from `fg.OutputTimeBase(0)` whenever the encoder's upstream is a `FilterGraph`. Surfaced by the `vf_scale_named_x264_no_audio` round-trip case.

### Added
- **Per-output `-shortest` / `-fs` and output-side `-ss`/`-t`/`-to` with `-copyts` (`Output.Shortest` / `Output.MaxFileSize` / `Config.CopyTS`).** Three new fields land the canonical "stop at the shortest stream", "cap output at N bytes", and "trim at the muxer with optional absolute-PTS preservation" pattern that shows up in essentially every overlay / music-video / silent-video-plus-audio job, every capture-with-quota pipeline, and every broadcast-PTS workflow. Implementation mirrors the FFmpeg 8.0 reference end-to-end: `Output.Shortest` is enforced per-output (mirrors the per-`OutputFile` sync queue set up in `fftools/ffmpeg_mux_init.c`) by recording the PTS at which the first feeder channel of an output closes and then draining-and-dropping further packets on the remaining channels of the same output; `Output.MaxFileSize` mirrors `mux->limit_filesize` from `fftools/ffmpeg_mux.c::mux_packet_filter` by querying `avio_tell` on the muxer's IO context before every `WritePacket` and stopping with a clean trailer once the limit is reached (new `av.OutputFormatContext.BytesWritten` wraps the C call); output-side `-ss`/`-t`/`-to` from `Output.Options` resolve through the new `pipeline.resolveOutputTiming` (parallel to the input-side helper) and are enforced in `pipeline/handlers.go::handleSink` by dropping packets below `start_time` and stopping muxing at `start_time + recording_time`, mirroring `fftools/ffmpeg_mux.c::of_streamcopy` and `fftools/ffmpeg_enc.c::check_recording_time`. `Config.CopyTS` is a global flag mirroring `-copyts`: when true the demuxer-side `ts_offset` shift normally applied after `-ss` (see `fftools/ffmpeg_demux.c`) is suppressed so the original timestamps are preserved end-to-end, and the meaning of any output-side `-ss`/`-to` flips from "offset from input start" to "absolute timeline values" (`stopTimestampUS(true)` anchors at `startUS + recordingUS` rather than at `recordingUS`). To avoid duplicating the six new policy checks across the single-channel and multi-channel paths, `handleSink` was refactored around a shared `processOne` helper that runs the checks in FFmpeg-faithful order (max-frames → output `-ss` drop → output `-t`/`-to` stop → shortest cap → shift back by `startUS` when `!copyTS` → file-size cap → write); a `sync.Mutex`-guarded `shortestPTSus` and an `atomic.Bool` `stopAll` propagate the per-output stop signal across the goroutines that drive the channels of a multi-stream output. `compat/ffcli` parses `-shortest`, `-fs N` (strict non-negative integer; rejects `-fs -1`, `-fs banana`), and the global `-copyts`. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `shortest` + `max_file_size`, top-level `copy_ts`); mirrored in `frontend/src/lib/jobTypes.ts`. Capability registry bumped (`-shortest`: missing → covered; `-fs`: missing → covered; output-side `-ss`/`-t`/`-to`: partial → covered; `-copyts` split out from the muxdelay-cluster entry into its own `covered` row pointing at `pipeline.Config.CopyTS`); roadmap §2.5 matrix flipped (3 cells to ✅, the muxdelay-cluster row to ⚠️ noting the remaining `-muxdelay`/`-muxpreload`/`-start_at_zero`/`-avoid_negative_ts` gap). Third Wave 1 item from §6 of the parity plan ([docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md)). Regression coverage: [pipeline/output_trim_test.go](pipeline/output_trim_test.go) (`TestResolveOutputTiming_FFmpegSemantics`, `TestOutputTiming_StopAndStartSemantics` covering the copyTS-on/-off stop-PTS arithmetic, `TestValidateMaxFileSizeRejectsNegative`) and [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go) (`TestParseShortest`, `TestParseFS`, `TestParseFSRejectsInvalid`, `TestParseCopyTS`). Out of scope for this PR: `shortest_buf_duration` tuning, `-fs` SI suffix parsing (`1M`, `500K`), and the rest of the muxdelay cluster (`-muxdelay`, `-muxpreload`, `-start_at_zero`, `-avoid_negative_ts`).
- **Per-output audio resync compensation (`Output.AudioSync` / FFmpeg `-async N`).** New optional integer field on `Output` mirrors the legacy FFmpeg `-async N` flag — removed from the FFmpeg 8.0 CLI in favour of `-af aresample=async=N`, but still in widespread use across community scripts and integration playbooks where audio drifts against video on VFR or capture sources. `0` (default) is off; `1` corrects only the start of the stream so the first sample lands at PTS 0 (rendered as `aresample=async=1:first_pts=0`); `N>1` enables continuous soft compensation up to `N` samples/sec (rendered as `aresample=async=N`). Underlying engine is libswresample's `swr_next_pts` → `swr_inject_silence` / `swr_drop_output` (hard) and `swr_set_compensation` (soft) — see [libswresample/swresample.c](../ffmpeg/libswresample/swresample.c). Plumbing lives in a new splice pass `pipeline.spliceAudioSyncForOutputs` ([pipeline/handlers.go](pipeline/handlers.go)) that runs after `spliceAudioAdaptersForEncoders` and injects an `__async__*` aresample filter node directly upstream of any existing `__aspl__` aformat+asetnsamples adapter (and otherwise immediately upstream of the audio encoder). Placement matters: aresample's sample injection/drop invalidates asetnsamples' exact-frame-size guarantee, so resync must happen before the adapter, not after. `compat/ffcli` accepts the legacy `-async N` flag with strict integer validation (rejects `-async -3`, `-async banana`); covered in `TestParseAsyncFlag` / `TestParseAsyncRejectsInvalid`. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json`; mirrored in `frontend/src/lib/jobTypes.ts`. Capability registry bumped from `missing` to `covered`; roadmap §2.4 matrix flipped to ✅. Second Wave 1 item from §6 of the parity plan ([docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md)). New regression coverage in [pipeline/audio_sync_test.go](pipeline/audio_sync_test.go) (5 tests: off-by-default; `N==1` renders `first_pts=0`; `N>1` renders continuous; video-only outputs untouched; validator rejects negatives).
- **Per-output frame-rate enforcement (`Output.FPSMode` / FFmpeg `-fps_mode` + legacy `-vsync`).** New optional `fps_mode` field on `Output` accepts `passthrough` (default), `vfr`, `cfr`, or `drop`, mirroring the FFmpeg flag that is the single biggest cause of HLS/DASH player A/V drift complaints. Per-frame renumber/drop/duplicate logic lives in [pipeline/fps_mode.go](pipeline/fps_mode.go) (`fpsRewriter`) and is consumed by `handleEncoder` for video streams only — audio/subtitle encoders see an empty mode and skip the rewriter entirely. `cfr` renumbers PTS to constant 1/framerate intervals, duplicating frames into forward gaps and dropping frames that arrive within half a frame duration of the next slot; `vfr` drops only non-monotonic frames; `drop` additionally drops near-duplicates within half a frame duration of the previous emission. Frame duration in the encoder time-base is derived from the new `EncoderContext.FrameRate()` / `EncoderContext.MediaType()` accessors in [av/encode.go](av/encode.go); when either is unavailable the rewriter degrades safely to passthrough. `compat/ffcli` parses `-fps_mode` directly and rewrites the legacy `-vsync` aliases (`0`/`auto`/`-1` → passthrough, `1` → cfr, `2` → vfr, `drop` → drop, named values pass through) so existing scripts round-trip without edits. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json`; mirrored in `frontend/src/lib/jobTypes.ts`. Capability registry bumped from `missing` to `covered`; roadmap §2.4 matrix flipped to ✅. First Wave 1 item from §6 of the parity plan. (Audio resync via `-async N` remains a separate Wave 1 follow-up.)

- **Container metadata + chapters end-to-end (`-metadata`, `-map_metadata`, `-map_chapters`).** Outputs gained `Metadata map[string]string` (container-level KV pairs, mirrors `ffmpeg -metadata key=value`) and `Chapters []Chapter` (explicit chapter table with seconds-based `Start`/`End` and a free-form `Metadata` map per entry). Inputs gained the matching `MapMetadata bool` / `MapChapters bool` switches that copy the input's container metadata / chapter table onto every output that does not set its own. Resolution rules match FFmpeg: an output-side `metadata` or `chapters` value fully replaces any mapped data (FFmpeg applies `-metadata` after `-map_metadata`); when more than one input opts in, metadata merges in declaration order (last writer wins per key) while chapters honour the first-input-wins single-source semantics of `-map_chapters`. Wiring lives in `pipeline/handlers.go.openSink`, applied between stream creation and `WriteHeader` so muxers (matroska, mp4, ogg, ffmetadata, …) actually honour the values. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (new `$defs/chapter`); mirrored in `frontend/src/lib/jobTypes.ts` (new `Chapter` interface). Round-trip coverage in [pipeline/metadata_test.go](pipeline/metadata_test.go) drives a stream-copy job into matroska with `metadata: {title, artist}` + two chapters and `ffprobe`s the result. With `kind: "lavfi"` the input `url` is interpreted as a filtergraph spec passed straight to libavfilter via the `lavfi` virtual demuxer (FFmpeg's `-f lavfi`), unlocking generators like `anullsrc`, `sine`, `color`, `testsrc`, `mandelbrot`, `smptebars`, etc. without needing an on-disk file. - **`Output.MaxFramesVideo` / `Output.MaxFramesAudio` (FFmpeg `-frames:v` / `-frames:a`).** New optional integer fields on the `Output` config let a job cap the number of muxed video / audio packets per output, mirroring FFmpeg's `-frames:v N` / `-vframes N` and `-frames:a N` / `-aframes N` flags. Enforcement lives in `pipeline/handlers.go.handleSink`: a per-channel `written` counter is incremented as packets are written; once the limit is reached the sink calls `pkt.Close()` and continues draining the channel without writing (so upstream encoders never block on a full channel and the trailer is still written cleanly when the input closes naturally). Counts post-encoder packets — matches ffmpeg's semantics for filter graphs that drop frames (`select=gt(scene,...)`, `select=not(mod(n,N))`, `tile=…`). Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json`; mirrored in `frontend/src/lib/jobTypes.ts`. Five new community-script fixtures exercise the plumbing: [`testdata/community-scripts/21_extract_frame.json`](testdata/community-scripts/21_extract_frame.json) (single JPEG via `mjpeg` muxer + `MaxFramesVideo: 1`), [`22_tile_thumbnails.json`](testdata/community-scripts/22_tile_thumbnails.json) (`select`+`scale`+`tile` contact sheet), [`23_scene_images.json`](testdata/community-scripts/23_scene_images.json) (first scene-change frame), [`24_scene_cut.json`](testdata/community-scripts/24_scene_cut.json) (re-encode with `-ss`/`-t` + `MaxFramesVideo` cap), and [`25_scene_cut_to.json`](testdata/community-scripts/25_scene_cut_to.json) (`-ss`/`-to` + `MaxFramesAudio` cap).
- **`xfade` / `acrossfade` / multi-clip transitions now run end-to-end.** `av.FilterPadConfig` (complex-filtergraph inputs) and `av.VideoFilterGraphConfig` (simple-filter chains) gained `FRNum, FRDen` fields, and the C `make_video_src_args` helper now appends `:frame_rate=N/D` to the buffersrc args string when the rate is known. `pipeline/handlers.go.createFilter` populates these from `av.StreamInfo.FrameRate`, and `resolveEdgeStreamInfo` re-queries `av_buffersink_get_frame_rate` / `av_buffersink_get_time_base` after each upstream filter so a `fps=24` node correctly propagates 24/1 (and the new time base) to the next stage. New `av.FilterGraph.OutputFrameRate` / `OutputTimeBase` accessors expose the buffersink-side values. Same plumbing also unblocks `framerate` and `minterpolate` in any complex graph. `handleFilter` now treats `EAGAIN` / `EOF` from `PushFrameAt` as "this input pad is drained" instead of failing the run, mirroring `fftools/ffmpeg_filter.c` — required because xfade closes its second input as soon as the transition window completes. The community-scripts skip guard around `13_xfade.json` and `14_crossfade_clips.json` is removed; both fixtures now pass. (Naming note: chose `FRNum`/`FRDen` to match the existing `TBNum`/`TBDen` / `SARNum`/`SARDen` shorthand on the same struct, in preference to the verbose `FrameRateNum`/`Den` mentioned in the roadmap.)
- **FFmpeg Coverage Roadmap** ([docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md)) — capability-based gap analysis between MediaMolder and the full FFmpeg command-line surface, structured as a six-phase plan (community-scripts gaps → universal mapper → output-side fidelity → broadcast/live → GUI completeness → conformance suite). Anchored on the new `testdata/community-scripts/` corpus (16/20 jobs running, 4 skipped, 15/35 not yet expressible) plus a second corpus of production-pattern command lines (animated `drawtext` expressions, two-pass `loudnorm`, multi-resolution split-and-mux ABR, multi-device hardware graphs, HDR `zscale`/`tonemap`, `minterpolate`, RNNoise speech denoise, lossless intermediates, raw-stream inputs). Each FFmpeg flag/feature is traced to a schema field or code location and marked covered/partial/missing. Linked from the README docs index.
- **Inspector: Timing controls on inputs and outputs.** Both the Input and Output forms now expose `-ss` (Start), `-t` (Duration) and `-to` (End) fields backed by the existing `Input.Options` / `Output.Options` JSON maps. Mirrors FFmpeg's positional semantics — placed on the input these flags restrict the demuxer (e.g. trim source to 30s before any further processing); placed on the output they restrict the muxer (trim the encoded result). Accepts seconds (`30`) or `HH:MM:SS[.ms]`. Round-trips through `compat/ffcli` so commands like `ffmpeg -ss 5 -t 30 -i in.mp4 -c copy out.mp4` populate the Inspector correctly.
- **Runtime: input-side `-ss` / `-t` / `-to` honoured by the demuxer** as a faithful Go port of `fftools/ffmpeg_demux.c`. `-ss` triggers an `avformat_seek_file(ctx, -1, INT64_MIN, target, target, 0)` (with the container's reported `start_time` added when known); `-t` / `-to` resolve through the same conflict rules ffmpeg.c uses (mutually exclusive — `-t` wins with a warning; `-to` alone converts to `recording_time = to - max(ss, 0)`; `-to <= -ss` is rejected); the demux loop then enforces `recording_time` per packet, mirroring the `dts >= recording_time + start_time` check in `input_packet_process()`. Every emitted packet's PTS/DTS is also shifted by `ts_offset = -timestamp` to mirror `ts_fixup()` in non-`copy_ts` mode, so a `-ss 450 -t 10` stream-copy produces a 10-second file (PTS [0, 10s]) instead of a 460-second container with a 10-second hole at the front. Time strings are parsed by binding `av_parse_time(..., 1)` exactly as the FFmpeg CLI does, so any value the FFmpeg command line accepts (`30`, `5.5`, `00:01:23.250`, `90s`) Just Works. Implemented in [pipeline/timing.go](pipeline/timing.go), with the seek + per-packet stop check + ts_offset shift wired into [pipeline/handlers.go](pipeline/handlers.go); new `av.ParseTime` / `av.InputFormatContext.StartTime` / `av.NoPTSValue` / `av.Packet.ShiftTS` / `av.Packet.SetPTS` / `av.Packet.SetDTS` exposed in [av/demux.go](av/demux.go) and [av/packet.go](av/packet.go). Regression coverage in [pipeline/timing_test.go](pipeline/timing_test.go).
- **Stream-copy nodes (`type: "copy"`).** A new node category lets a graph forward demuxer `AVPacket`s straight to the muxer with no decode/encode — the standard "swap container, keep streams" pattern. One copy node per media type (`copy_video`, `copy_audio`, `copy_subtitle`, `copy_data`) is exposed under the new **Copy** palette section between Encoders and Processors. The implicit-encoder splice treats copy nodes as already-encoded, so a manual `in:a:0 → copy_audio → out:a` edge no longer gets an AAC encoder spliced in. The runtime adds the output stream via `avcodec_parameters_copy` (no encoder context required), rescales packet timestamps from the demuxer's `time_base` to the muxer's post-header `time_base`, and clones packets when one demuxed stream feeds multiple copy nodes. Implemented across `graph/graph.go` (new `KindCopy`), `av/packet.go` (`Packet.Rescale`, `ClonePacket`), `av/mux.go` (`AddStreamFromInput`, `StreamTimeBase`), `pipeline/handlers.go` (new `handleCopy` + per-stream-type packet routing in `handleSource` + per-channel rescale in `handleSink`), `internal/gui/api.go` (palette entries + new `"Copy"` category in `categoryOrder`), and `frontend/src/lib/spawn.ts` / `MMNode.tsx` / `jsonAdapter.ts`. Documented in [docs/json-config-reference.md](docs/json-config-reference.md) and [docs/gui.md](docs/gui.md).
- **ABR ladder example** (`testdata/examples/35_abr_ladder.json`) — single input fans out to four `scale` filters (1920×1080, 1280×720, 960×540, 640×360), each feeds a dedicated `libx264` encoder at 7000/4000/2000/1000 kbps, with a single AAC audio encoder fanned out to all four MP4 muxers. Exercises explicit fan-out, multi-output mux, and per-encoder bitrate independence end-to-end.

### Changed
- **Schema v1.2 is now accepted by the runtime validator.** `pipeline.Config.validate` previously hard-rejected `"1.2"` even though the GUI emits it for every newly-saved job. v1.2 currently adds nothing to the schema beyond `graph.ui.positions` (already accepted) and is forward-compatible with v1.0/v1.1.
- **Encoder parameters are forwarded verbatim to libavcodec.** Every key in a node's `params` that is not a runtime-reserved field (`codec`, `width`, `height`, `bitrate`, `threads`, `thread_type`) is now passed through `EncoderOptions.ExtraOpts → av_dict_set → avcodec_open2`, so codec-specific options written by the GUI (`preset`, `crf`, `maxrate`, `bufsize`, `x264-params`, …) actually reach the encoder. `b` is recognised as an alias for `bitrate`, and `g` maps to `EncoderOptions.GOPSize`. Covered by `TestCollectEncoderExtraOpts` in `pipeline/handlers_test.go`.
- **GUI: ABR audio encoded once and fanned out.** `35_abr_ladder.json` now defines a single explicit `enc_audio` (codec `aac`, 128 kbps) with edges to all four output muxers, instead of four redundant implicit AAC encoders.
- **GUI: `libx264` / `libx265` `preset` rendered as a dropdown** with the canonical 10 entries (`ultrafast` … `placebo`, default `medium`). The structure is generic — add to `ENCODER_OPTION_CHOICES` in `frontend/src/lib/encoderSchema.ts` to surface other codec presets the same way (e.g. libvpx_vp9 `deadline`).

### Fixed
- **GUI: Imported stream-copy nodes now match user-spawned ones.** Materialised copy nodes (synthesised by `materializeImplicitEncoders` from a `codec: "copy"` output) used `auto_enc_<output>_<type>` ids and rendered with all four media-type pins, so a `ffmpeg -c copy` import looked nothing like the `Copy (video)` / `Copy (audio)` cards from the palette. The materialiser now emits canonical `copy_<type>_<output>` ids, `nodeDisplayLabel` recognises them and renders the friendly `copy_<type>` heading, and `configToFlow` infers the supported media type from the node's incident edges so MMNode's `describeKind` shows "Video stream copy" / "Audio stream copy" with a single video- or audio-only pin per side. Visual parity with palette-spawned copy nodes restored.
- **GUI Import FFmpeg dialog: stream-copy commands no longer crash.** `POST /api/convert-cmd` produced a config whose `graph.nodes` was the JSON literal `null` (Go's nil-slice marshalling) for any command without filters or explicit encoders — e.g. `ffmpeg -i in -c copy -t 30 out`. The frontend then crashed in `materializeImplicitEncoders` calling `.map()` on null. The `compat/ffcli` parser now always emits non-nil empty `Nodes` / `Edges` slices, and `materializeImplicitEncoders` defensively coerces null array fields. Regression tests added in [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go).
- **`compat/ffcli`: `-t`, `-ss`, `-to` are now preserved.** Previously these timing flags were consumed by the catch-all `-flag value` branch and dumped into an unused `globalOpts` bucket, silently dropping them. They are now attached as `Options` entries on the next file specifier on the command line (input or output), matching FFmpeg's positional semantics — so `ffmpeg -i in -c copy -t 30 out` round-trips with `outputs[0].options.t = "30"` and the demuxer/muxer pass-through honours it.
- **GUI: CRF / QP defaults no longer show as `-1`.** libavcodec encodes "unset, pick the codec's built-in default" as the sentinel `(default=-1, min=-1, max=FLT_MAX)` for CRF/QP-style options on libx264/libx265/libsvtav1/libvpx_vp9/libaom_av1. The Inspector now substitutes a per-codec override table (`ENCODER_PRIMARY_OVERRIDES`) so the placeholder reads `(default — 23)` and the slider/number bounds are realistic (e.g. libx265 CRF 0–51 with default 28).
- **GUI: edges serialise using the node id, not the display label.** `deriveEndpoint` was returning `node.data.label` for graph nodes, which is the codec/filter display name (e.g. `aac`, `libx264`); a hand-wired `enc_audio` was being saved as `to: "aac"` and the runtime then failed with `unknown destination node aac`. The graph-node branch now emits the React Flow id (which equals the job-side id).
