## [Unreleased]

### Fixed
- **Positional FFmpeg-style filter args (`scale=320:240`) reach libavfilter intact.** `compat/ffcli/ffcli.go.parseFilterExpr` records each `:`-separated token without an embedded `=` as `_pos0`, `_pos1`, ... so the original order can be recovered, but `pipeline/engine.go.buildFilterSpec` previously emitted those keys verbatim, producing `scale=_pos0=320:_pos1=240` which `avfilter_graph_parse_ptr` rejected with `Could not set non-existent option '_pos0'`. The builder now partitions params into positional (`_posN`) and named, sorts positional by index, and emits them as bare in-order values before any named args. Surfaced by the `vf_scale_positional_x264_audio_copy` round-trip case.
- **libx264 (and any encoder) fed by a filter graph no longer blows up the container duration.** When the upstream filter graph's buffersink advertised a finer `time_base` than `1/framerate` (e.g. demuxer TB `1/12288` propagated through `scale`), the encoder was opened with `time_base = 1/framerate` only, so frame PTS produced in the buffersink's units were reinterpreted in the encoder's coarser units — a 24 fps clip ran out at ~512× the real duration (219 frames → ~4650 s instead of ~9.1 s). `av.EncoderOptions` gained a `TimeBase [2]int` field that, when set, overrides the framerate-derived default; `pipeline/handlers.go.createEncoder` now populates it from `fg.OutputTimeBase(0)` whenever the encoder's upstream is a `FilterGraph`. Surfaced by the `vf_scale_named_x264_no_audio` round-trip case.

### Added
- **Forced keyframes (`Output.ForceKeyFrames` / FFmpeg `-force_key_frames SPEC`).** New optional string field on `Output` lands the canonical `-force_key_frames "expr:gte(t,n_forced*2)"` GOP-control idiom that every HLS / DASH segmenter recipe documents and that, missing, silently produces broken playlists (segment boundaries that don't fall on IDR frames). Three FFmpeg-faithful grammars accepted by `pipeline/force_key_frames.go::parseForceKeyFrames` (validated at config-load so a malformed spec is rejected up-front, never at first frame): `expr:EXPR` (libavutil expression evaluated per video frame; vars `n` / `n_forced` / `prev_forced_n` / `prev_forced_t` / `t` mirror [ffmpeg.h:557-561](../ffmpeg/fftools/ffmpeg.h)), `source` (copy keyframes from input — fires when the upstream decoded frame's `pict_type` is I), or comma-separated float-second time list (`3.0,7.5,10.25`; sorted ascending; HH:MM:SS form deferred). Per-encoder runtime state lives in `forceKeyFramesMatcher`, built once in `pipeline/handlers.go::handleEncoder` from the encoder's actual time-base so `t = pts * num / den` matches the encoder's view of time; the expression is compiled once via the new `av.ParseExpression` wrapper around `av_expr_parse` so the per-frame hot loop only does an `av_expr_eval` (cgo cost: ~50ns/frame). The matcher is consulted in `sendOne` exactly once per frame in PTS order — including each duplicated emission from the CFR fps_mode forward-gap fill, mirroring FFmpeg's per-frame `forced_kf_apply` invocation in [`fftools/ffmpeg_enc.c`](../ffmpeg/fftools/ffmpeg_enc.c) line 738. On a match the matcher stamps `frame.pict_type = AV_PICTURE_TYPE_I` via the new `av.Frame.SetPictType` setter (libavcodec honours that as an IDR request regardless of the encoder's GOP cadence — faithful port of `forced_kf_apply` lines 738-776). Counters (`n` / `n_forced` / `prev_forced_n` / `prev_forced_t`) advance on every call so the expression sees the post-rewrite PTS stream the encoder actually receives. New av-layer surface: [`av/frame.go`](av/frame.go) gains `Frame.PictType()` + `Frame.SetPictType(int)` accessors and the `PictureTypeNone` / `I` / `P` / `B` / `S` / `SI` / `SP` / `BI` constants (0..7) mirroring the `AVPictureType` enum; [`av/expr.go`](av/expr.go) gains `ParsedExpression` (`ParseExpression(expr, names) → *ParsedExpression`, `Eval([]float64) → float64`, `Close()`) wrapping `av_expr_parse` + `av_expr_eval` + `av_expr_free` for the repeated-evaluation case the per-frame matcher needs (the existing `EvalExpression` one-shot helper used by the GUI's filter validator stays untouched). Wired through `expandImplicitEncoders` ([pipeline/handlers.go](pipeline/handlers.go)) which propagates `Output.ForceKeyFrames` onto the synthetic encoder node as `__force_key_frames` (only on video edges; audio/subtitle encoders see no marker and skip the matcher entirely). `compat/ffcli` parses `-force_key_frames SPEC` end-to-end and latches the value onto the next output. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `force_key_frames: string`); mirrored in `frontend/src/lib/jobTypes.ts`. - **Tee muxer fan-out (`Output.Kind = "tee"` + `Output.Targets []TeeTarget` / FFmpeg `-f tee "[opts]url1|[opts]url2"`).** Encode once, mux to N container slaves in a single process — the canonical "ABR delivery" / "MP4 + HLS + DASH at the same time" / "RTMP egress + local archive" pattern. Routes the encoded packet stream through libavformat's built-in `tee` muxer ([`libavformat/tee.c`](../ffmpeg/libavformat/tee.c)) so neither the filter graph nor the encoder runs more than once per stream. Each `TeeTarget` carries a required `URL` plus the four typed slave-AVOption fields documented in `tee_options[]` — `Format` (`f`, the slave muxer name), `Select` (a stream specifier filter applied per-slave), `BSFs` (a comma-separated bitstream-filter chain, e.g. `h264_mp4toannexb` for the HLS leg), `OnFail` (`abort` | `ignore`, default `abort`), `UseFifo` (wrap the slave in libavformat's `fifo` muxer for non-blocking writes), and `FifoOptions` (the inner fifo's option string) — plus a free-form `Options` map for any obscure tee-slave knob the typed fields don't promote. New `pipeline.buildTeeSlavesURL` renders the `[k=v:k=v]url|[k=v]url` slave-list grammar deterministically (typed fields first in canonical order, then `Options` alphabetised) with `:`/`]`/`|`/`\` correctly backslash-escaped per `av_get_token`. New `av.OpenTeeOutput` calls `avformat_alloc_output_context2(&ctx, nil, "tee", slavesURL)` and skips the `.tmp`-suffix-plus-atomic-rename dance the file path uses (tee is `AVFMT_NOFILE` — slaves manage their own files). - **Per-input demuxer pacing, looping, and timestamp offset (`Input.StreamLoop` / `Input.ITSOffset` / `Input.ReadRate` / FFmpeg `-stream_loop` + `-itsoffset` + `-re`/`-readrate`).** Three demuxer-side typed fields land the most-requested per-input controls. `Input.StreamLoop` (0=off, N>0=play N+1 times, -1=infinite) mirrors FFmpeg's `-stream_loop` — `pipeline/handlers.go::handleSource` tracks per-iteration `(min_pts, max_pts)` of demuxed packets in AV_TIME_BASE microseconds, and on EOF, if loops remain, calls `SeekFile(input.StartTime())`, accumulates `(max - min)` into a new `loopOffsetUS` field, decrements the counter (unless it is -1, the infinite sentinel), and shifts every subsequent packet's PTS/DTS by the accumulated cycle duration so timestamps remain monotone across iterations. Faithful port of `fftools/ffmpeg_demux.c::seek_to_start` + the `pkt->pts += duration` arithmetic in `ts_fixup`. Canonical use case: a 5 s logo PNG looped indefinitely against a finite main video with `-shortest` ending the output. `Input.ITSOffset` (seconds, may be negative) mirrors `-itsoffset`; `openSource` composes it additively into the existing `tsOffsetUS` field — exactly as FFmpeg composes `f->ts_offset = o->input_ts_offset - timestamp` in `fftools/ffmpeg_demux.c` — so it stacks cleanly with the implicit `-ss` ts_offset and reaches every demuxed packet via the same `Packet.ShiftTS` path. Sub-millisecond resolution preserved end-to-end (microseconds). Unblocks A/V slip correction on dubbed sources (`-itsoffset -0.030 -i dub.wav`). `Input.ReadRate` (1.0 = realtime; 0 = unpaced default) plus `ReadRateInitialBurst` (default 0.5 s when ReadRate is non-zero) and `ReadRateCatchup` (default ReadRate × 1.05) mirror `-re` / `-readrate` / `-readrate_initial_burst` / `-readrate_catchup`. Implementation lives in a new [`pipeline/read_rate.go`](pipeline/read_rate.go) `readRatePacer` — a faithful port of `fftools/ffmpeg_demux.c::readrate_sleep` including the same 0.3 s lag-detection threshold and the catch-up-multiplier recovery state machine. The pacer is invoked per packet from `handleSource` after the PTS shifts (so it compares wallclock-elapsed against post-offset PTS, matching FFmpeg's `ds->dts` use). Sleeps via a context-aware `time.Timer` so pipeline cancellation aborts pacing immediately. Required for live-restream / RTMP / SRT egress and any HLS/DASH push that relies on segment walltime equalling media time. New av-layer surface: [`av/packet.go`](av/packet.go) gains `Packet.Duration()` (used by the loop's per-iteration max-PTS tracker to advance to the *end* of the last packet rather than its start). `compat/ffcli` parses `-stream_loop N` (rejects values < -1, the FFmpeg-undefined region — only -1 is the infinite sentinel), `-itsoffset T` (bare seconds form; the HH:MM:SS form `av_parse_time` accepts is deferred until a corpus job needs it), `-re`, `-readrate FACTOR`, `-readrate_initial_burst SECS`, and `-readrate_catchup FACTOR`. All five flags latch onto the *next* `-i` (matches FFmpeg's `OPT_INPUT | OPT_OFFSET` storage on `InputFile`), so `-stream_loop -1 -i logo.png -i main.mp4` correctly loops `logo.png` only — covered by `TestParseDemuxerFlagsLatchToNextInput`. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json`; mirrored in `frontend/src/lib/jobTypes.ts`. - **Per-stream metadata + disposition (`Output.Streams []StreamSpec` / FFmpeg `-metadata:s:<type>:<idx>` + `-disposition:s:<type>:<idx>`).** New optional `Streams` slice on `Output` lets a job tag individual output streams with key/value metadata (canonical use cases: `language=eng` / `language=fra` on dual-language audio, `title=Director Commentary` on a secondary audio track, `language=eng` + `disposition=default+forced` on a forced-narrative subtitle) and toggle the AV_DISPOSITION_* bitfield (default, forced, hearing_impaired, visual_impaired, captions, lyrics, comment, dub, original, attached_pic, ...). Each `StreamSpec` carries `Type` (`v`/`a`/`s`/`d`), 0-based `Index` within that media type, an optional `Metadata` map, and an optional `Disposition` string. Resolution counts streams of the requested media type in the order they were added to the muxer — the same convention FFmpeg's `cmdutils.c::check_stream_specifier` uses for `s:<type>:<idx>` — so `{Type:"a", Index:1}` resolves to the second audio AVStream regardless of how video / audio edges were declared on the sink. Implementation mirrors the FFmpeg 8.0 reference end-to-end: `Output.Shortest` is enforced per-output (mirrors the per-`OutputFile` sync queue set up in `fftools/ffmpeg_mux_init.c`) by recording the PTS at which the first feeder channel of an output closes and then draining-and-dropping further packets on the remaining channels of the same output; `Output.MaxFileSize` mirrors `mux->limit_filesize` from `fftools/ffmpeg_mux.c::mux_packet_filter` by querying `avio_tell` on the muxer's IO context before every `WritePacket` and stopping with a clean trailer once the limit is reached (new `av.OutputFormatContext.BytesWritten` wraps the C call); output-side `-ss`/`-t`/`-to` from `Output.Options` resolve through the new `pipeline.resolveOutputTiming` (parallel to the input-side helper) and are enforced in `pipeline/handlers.go::handleSink` by dropping packets below `start_time` and stopping muxing at `start_time + recording_time`, mirroring `fftools/ffmpeg_mux.c::of_streamcopy` and `fftools/ffmpeg_enc.c::check_recording_time`. `Config.CopyTS` is a global flag mirroring `-copyts`: when true the demuxer-side `ts_offset` shift normally applied after `-ss` (see `fftools/ffmpeg_demux.c`) is suppressed so the original timestamps are preserved end-to-end, and the meaning of any output-side `-ss`/`-to` flips from "offset from input start" to "absolute timeline values" (`stopTimestampUS(true)` anchors at `startUS + recordingUS` rather than at `recordingUS`). - **Schema v1.2 is now accepted by the runtime validator.** `pipeline.Config.validate` previously hard-rejected `"1.2"` even though the GUI emits it for every newly-saved job. v1.2 currently adds nothing to the schema beyond `graph.ui.positions` (already accepted) and is forward-compatible with v1.0/v1.1.
- **Encoder parameters are forwarded verbatim to libavcodec.** Every key in a node's `params` that is not a runtime-reserved field (`codec`, `width`, `height`, `bitrate`, `threads`, `thread_type`) is now passed through `EncoderOptions.ExtraOpts → av_dict_set → avcodec_open2`, so codec-specific options written by the GUI (`preset`, `crf`, `maxrate`, `bufsize`, `x264-params`, …) actually reach the encoder. `b` is recognised as an alias for `bitrate`, and `g` maps to `EncoderOptions.GOPSize`. Covered by `TestCollectEncoderExtraOpts` in `pipeline/handlers_test.go`.
- **GUI: ABR audio encoded once and fanned out.** `35_abr_ladder.json` now defines a single explicit `enc_audio` (codec `aac`, 128 kbps) with edges to all four output muxers, instead of four redundant implicit AAC encoders.
- **GUI: `libx264` / `libx265` `preset` rendered as a dropdown** with the canonical 10 entries (`ultrafast` … `placebo`, default `medium`). The structure is generic — add to `ENCODER_OPTION_CHOICES` in `frontend/src/lib/encoderSchema.ts` to surface other codec presets the same way (e.g. libvpx_vp9 `deadline`).

### Fixed
- **GUI: Imported stream-copy nodes now match user-spawned ones.** Materialised copy nodes (synthesised by `materializeImplicitEncoders` from a `codec: "copy"` output) used `auto_enc_<output>_<type>` ids and rendered with all four media-type pins, so a `ffmpeg -c copy` import looked nothing like the `Copy (video)` / `Copy (audio)` cards from the palette. The materialiser now emits canonical `copy_<type>_<output>` ids, `nodeDisplayLabel` recognises them and renders the friendly `copy_<type>` heading, and `configToFlow` infers the supported media type from the node's incident edges so MMNode's `describeKind` shows "Video stream copy" / "Audio stream copy" with a single video- or audio-only pin per side. Visual parity with palette-spawned copy nodes restored.
- **GUI Import FFmpeg dialog: stream-copy commands no longer crash.** `POST /api/convert-cmd` produced a config whose `graph.nodes` was the JSON literal `null` (Go's nil-slice marshalling) for any command without filters or explicit encoders — e.g. `ffmpeg -i in -c copy -t 30 out`. The frontend then crashed in `materializeImplicitEncoders` calling `.map()` on null. The `compat/ffcli` parser now always emits non-nil empty `Nodes` / `Edges` slices, and `materializeImplicitEncoders` defensively coerces null array fields. Regression tests added in [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go).
- **`compat/ffcli`: `-t`, `-ss`, `-to` are now preserved.** Previously these timing flags were consumed by the catch-all `-flag value` branch and dumped into an unused `globalOpts` bucket, silently dropping them. They are now attached as `Options` entries on the next file specifier on the command line (input or output), matching FFmpeg's positional semantics — so `ffmpeg -i in -c copy -t 30 out` round-trips with `outputs[0].options.t = "30"` and the demuxer/muxer pass-through honours it.
- **GUI: CRF / QP defaults no longer show as `-1`.** libavcodec encodes "unset, pick the codec's built-in default" as the sentinel `(default=-1, min=-1, max=FLT_MAX)` for CRF/QP-style options on libx264/libx265/libsvtav1/libvpx_vp9/libaom_av1. The Inspector now substitutes a per-codec override table (`ENCODER_PRIMARY_OVERRIDES`) so the placeholder reads `(default — 23)` and the slider/number bounds are realistic (e.g. libx265 CRF 0–51 with default 28).
- **GUI: edges serialise using the node id, not the display label.** `deriveEndpoint` was returning `node.data.label` for graph nodes, which is the codec/filter display name (e.g. `aac`, `libx264`); a hand-wired `enc_audio` was being saved as `to: "aac"` and the runtime then failed with `unknown destination node aac`. The graph-node branch now emits the React Flow id (which equals the job-side id).

### Added (earlier in this cycle)
- **Encoder Inspector — Advanced view, search, and Raw options.** The Inspector's encoder form now exposes every remaining AVOption under a collapsible **Advanced** section, bucketed by a lightweight heuristic (`Threading`, `Quality`, `Color`, `Motion`, `Profile / Level`, `GOP & frames`, `Other`). A search box filters across all groups by option name and help text. A separate **Raw options** section surfaces codec-native parameter strings (`x264-params`, `x264-opts`, `x265-params`, `svtav1-params`, `aom-params`, `vpx-params`) as multi-line text inputs that are passed through verbatim. Completes PR3 of 3 for the structured encoder configuration UI. Implemented entirely in `frontend/src/components/EncoderForm.tsx` (no new backend dependencies).
- **Encoder Inspector form (primary controls).** Selecting an encoder node now loads the codec's option schema from `/api/encoders/{name}/options` and renders typed controls for the four most common roles — Preset, Rate control, Quality (CRF/CQ/Q) **or** Bit rate (whichever applies), and Keyframe interval — using the codec's own ranges, defaults, and named values (`AV_OPT_TYPE_CONST` enums become dropdowns). Extra options can still be set under the existing key/value Params editor; the upcoming PR3 will add the full Advanced view, search, and raw-options escape hatches. Implemented in `frontend/src/lib/encoderSchema.ts` (typed fetcher + per-encoder UI-role table for libx264/libx265/libsvtav1/libvpx_vp9/libaom_av1/h264_nvenc/hevc_nvenc/videotoolbox/aac/libfdk_aac/libopus/libmp3lame), `frontend/src/components/controls/OptionControl.tsx`, and `frontend/src/components/EncoderForm.tsx`. Foundation for the upcoming structured Inspector form for encoder nodes (PR1 of 3). Implemented in `av/options.go` (cgo helpers + `EncoderInfo`/`EncoderOption`/`EncoderOptionEnum` types) and `internal/gui/encoders.go` (handler with in-memory caching).

The hover popover gained a **Delete** button. The endpoint of any existing connection can be grabbed and dragged to a different handle to re-route. Documented in `docs/gui.md` and the in-app Help dialog. Implemented in `frontend/src/app.tsx` (selectedEdgeIds + Backspace handler), `frontend/src/components/MMEdge.tsx` (Delete button via `useReactFlow().deleteElements`), and `frontend/src/styles.css` (.react-flow__edge.selected highlight).

Hover (or click to pin) any connection to open a popover at the midpoint listing every known attribute for the stream and which node established each value. The probe + inference pipeline now surfaces a much richer set of properties: codec profile/level, bit_rate, bit_depth, color_space, color_range, color_primaries, color_transfer, sample aspect ratio, field order, r_frame_rate, codec FourCC tag, and start time, in addition to the existing width/height/pix_fmt/frame_rate/sample_rate/sample_fmt/channels/channel_layout/duration. Image files are reported as a single video stream with the image's geometry and pixel format. Implemented in `frontend/src/components/MMEdge.tsx`, `frontend/src/lib/streamAttrs.ts`, `av/demux.go`, `av/probe_names.go` (new helpers `ColorSpaceName`, `ColorRangeName`, `ColorPrimariesName`, `ColorTransferName`, `ProfileName`, `FieldOrderName`, `CodecTagString`), and `internal/gui/probe.go`.
- **GUI: `Get properties` button on Input nodes** — clicking it in the Inspector calls a new `POST /api/probe` backend endpoint that opens the source URL with `avformat_open_input` + `avformat_find_stream_info` and returns per-stream technical metadata (codec, width/height, pix_fmt, frame rate, sample rate, sample format, channels, channel layout, duration). The probed values are stored on the Input node (editor-only — never serialised) and seed the edge attribute inference, so every downstream connection in the graph gets accurate technical attributes automatically. Implemented in `internal/gui/probe.go`, `av/probe_names.go` (helpers `PixFmtName`, `SampleFmtName`, `CodecName`, `DefaultChannelLayoutName`), and `frontend/src/components/Inspector.tsx`.
