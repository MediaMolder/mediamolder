### Added
- **Wave 6 muxer/encoder fidelity #33: per-output encoder timing / field order / interlaced encode.** Three new `Output` fields close the broadcast-grade encoder edge cases from §6.6 of [docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md). **`Output.EncoderTimeBase string`** mirrors FFmpeg per-stream `-enc_time_base` (`fftools/ffmpeg_mux_init.c` L1391-1417): accepts the sentinel strings `"demux"` (use the demuxer-side AVStream.time_base verbatim) and `"filter"` (use the upstream filtergraph's buffersink TB), or an explicit `"N/D"` rational (`"1/30000"` for NTSC drop-frame, `"1/25"` for PAL). **`Output.FieldOrder string`** ∈ {`""`, `"progressive"`, `"tt"`, `"bb"`, `"tb"`, `"bt"`} stamps `AVCodecContext.field_order` (libavcodec/defs.h::AVFieldOrder enum). **`Output.InterlacedEncode bool`** ORs `AV_CODEC_FLAG_INTERLACED_DCT | AV_CODEC_FLAG_INTERLACED_ME` (avcodec.h L310/L331) onto the encoder context — required for 1080i broadcast (XDCAM HD422, AVC-Intra, ProRes 422 HQ). New [pipeline/encoder_timing.go](pipeline/encoder_timing.go) hosts `parseEncoderTimeBase` (rejects bare integers / negative numerators / zero denominators) + `validateEncoderTiming` (rejects `EncoderTimeBase` on subtitle-only outputs / without an encoder; rejects unknown `FieldOrder` enum; rejects `FieldOrder`/`InterlacedEncode` without a video encoder; rejects the contradiction `InterlacedEncode=true` + `FieldOrder="progressive"`). Wired through `expandImplicitEncoders` ([pipeline/handlers.go](pipeline/handlers.go)) which propagates the three fields onto the synthetic encoder node as `__enc_time_base` / `__field_order` / `__interlaced` sentinels (only on video edges); `createEncoder` reads them back into `av.EncoderOptions{FieldOrder, InterlacedEncode, TimeBase}`. New `av.EncoderOptions.FieldOrder int` + `av.EncoderOptions.InterlacedEncode bool` are forwarded to the AVCodecContext in [av/encode.go](av/encode.go)::`OpenEncoder`. `compat/ffcli` parses `-enc_time_base SPEC`, `-field_order ORDER`, and decomposes `-flags +ildct+ilme` (the canonical FFmpeg way to enable interlaced encode) into `Output.InterlacedEncode = true` while still propagating the raw `-flags` value into the encoder AVOptions for completeness. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `encoder_time_base: string`, `field_order` enum, `interlaced_encode: boolean`); mirrored in [frontend/src/lib/jobTypes.ts](frontend/src/lib/jobTypes.ts). Roadmap §6.6 #33 ✅, §2.4 matrix `-enc_time_base` and field-order rows flipped to ✅, capabilities.yaml `-enc_time_base` + `-field_order` flipped to covered. Regression coverage: [pipeline/encoder_timing_test.go](pipeline/encoder_timing_test.go) (`TestParseEncoderTimeBase` 9 cases including sentinels / rational / rejects bare int + zero den; `TestValidateEncoderTiming` 10 cases including subtitle-only rejection + interlaced+progressive contradiction; `TestFieldOrderEnumValue`) and [compat/ffcli/encoder_timing_test.go](compat/ffcli/encoder_timing_test.go) (`TestParseEncoderTiming` round-trips all three flags in one command line; `TestParseEncoderTimingErrors` covers missing-arg paths). Out of scope for this PR: the `"demux"`/`"filter"` sentinels are accepted at parse + validate time but the runtime currently lets the existing buffersink TB precedence handle the propagation (the explicit `N/D` form does land on `AVCodecContext.time_base` via `EncoderOptions.TimeBase`); per-stream-spec timing overrides (`-enc_time_base:v:1`) — deferred until #30 lands the StreamSpec.Encoder override shape.

- **Wave 6 muxer/encoder fidelity #34: `Input.SubtitleCharenc` (`-sub_charenc`).** Promotes FFmpeg's per-input `-sub_charenc CODE` to a typed schema field. Routed through new `av.OpenSubtitleDecoderWithOptions(input, idx, av.SubtitleDecoderOptions{Charenc})` which sets `AVCodecContext.sub_charenc` (via `av_strdup`) before `avcodec_open2` so libavcodec/decode.c L860's iconv routes incoming SRT/ASS/WebVTT bytes through `Charenc → UTF-8` before the text-subtitle decoder sees them. Mirrors the FFmpeg-side `FF_SUB_CHARENC_MODE_PRE_DECODER` branch in libavcodec/decode.c L2014-2023. Setting the field on a bitmap-subtitle stream (PGS, DVB, DVD, …) is rejected at decoder-open with a clear error referencing the codec_id — the av-layer probes `AVCodecDescriptor.props & AV_CODEC_PROP_TEXT_SUB` via the new `av.SubtitleStreamProps` / `av.IsTextSubtitleProps` / `av.IsBitmapSubtitleProps` / `av.SubtitleEncoderProps` helpers so the same predicate is reusable from any future codec-pair validator. Forced / hearing-impaired subtitle flags continue to ride on the per-stream `Disposition` field shipped in Wave 1 #3 (no schema change needed). `compat/ffcli` parses `-sub_charenc CODE` and latches it onto the next `-i` only — mirrors FFmpeg's `OPT_INPUT` semantics so `-sub_charenc WINDOWS-1251 -i ru.srt -i en.srt` applies CODE to `ru.srt` only. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (input-level `subtitle_charenc: string`); mirrored in `frontend/src/lib/jobTypes.ts`. Roadmap §6.6 #34 ✅, §2.6 row flipped to ✅, capabilities.yaml `-sub_charenc` flipped to covered. Regression coverage: [compat/ffcli/input_demuxer_flags_test.go](compat/ffcli/input_demuxer_flags_test.go) `TestParseSubCharenc` proves the per-input latching semantics (CODE applies to first `-i` only, second `-i` gets a fresh latch). Out of scope for this PR: codec-pair validation (`mov_text` ↔ bitmap subtitle pair) is best done at runtime once the input codec is known; the av-layer helpers (`SubtitleStreamProps` + `SubtitleEncoderProps`) are the building blocks but no caller drives them yet — deferred until a corpus job needs the early failure.

- **Wave 6 muxer/encoder fidelity #32: per-output `-vn` / `-an` / `-sn` / `-dn`.** Promotes FFmpeg's per-output disable flags from "implied by which graph edges connect" to four typed `Output` booleans: `DisableVideo`, `DisableAudio`, `DisableSubtitle`, `DisableData`. Mirrors `fftools/ffmpeg_opt.c` L1977/2078/2115/2187 — the `OPT_OUTPUT` half of the dual-purpose disable bools (the same flag with `OPT_INPUT` drops streams at the demuxer instead). Wired into [pipeline/handlers.go](pipeline/handlers.go)::`configToGraphDef`: edges feeding a sink whose `Output` has the corresponding flag set are skipped before `expandImplicitEncoders` runs, so no implicit encoder is synthesised for the disabled media type and the stream-copy path never registers a copy stream at the sink. Validator rejects an output with all four flags set (would yield a zero-stream muxer that `avformat_write_header` refuses). Lets users build "audio-only re-encode" outputs without rewiring graph edges, and gives the `compat/ffcli` importer a typed home for the legacy flags. `compat/ffcli` parses `-vn`/`-an`/`-sn`/`-dn` (latched onto the next output URL) and additionally clears the legacy `codecV`/`codecA`/`codecS = "none"` selection so existing call sites keep working. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `vn`/`an`/`sn`/`dn` booleans); mirrored in `frontend/src/lib/jobTypes.ts`. Regression coverage: [pipeline/muxer_options_test.go](pipeline/muxer_options_test.go) (`TestValidateDisableAllStreamsRejected` covers the all-four-set guard; `TestConfigToGraphDefDropsDisabledEdges` proves audio edges are filtered while video edges survive) and [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go) (`TestParseDisableStreams` round-trips all four flags on the first output and verifies a second output gets a fresh latch).

- **Wave 6 muxer/encoder fidelity #29: per-output `muxdelay` / `muxpreload` / `avoid_negative_ts` + global `start_at_zero`.** Closes the timestamp-policy cluster started by `Config.CopyTS` in Wave 1 #2 (§6.6 of [docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md)). Three new optional fields on `Output` and one on `Config`. **`Output.MuxDelay float64`** (seconds) caps how far the muxer is allowed to buffer ahead of the latest demux/decode timestamp before flushing — mirrors FFmpeg `-muxdelay` (per-output `OPT_FLOAT`; [`fftools/ffmpeg_mux_init.c` L3447](../ffmpeg/fftools/ffmpeg_mux_init.c) — written as `oc->max_delay = muxdelay * AV_TIME_BASE`). **`Output.MuxPreload float64`** (seconds) pre-rolls the initial demux/decode delay window — mirrors FFmpeg `-muxpreload` ([`fftools/ffmpeg_mux_init.c` L3444](../ffmpeg/fftools/ffmpeg_mux_init.c) — emitted into the muxer AVDict as `preload = muxpreload * AV_TIME_BASE`); the historic consumer is `libavformat/mpegenc.c`. **`Output.AvoidNegativeTS string`** ∈ {`""`, `auto`, `disabled`, `make_non_negative`, `make_zero`} controls libavformat's automatic timestamp-shift policy at the muxer boundary — mirrors `-avoid_negative_ts` (AVFormatContext AVOption, [`libavformat/options_table.h` L95-99](../ffmpeg/libavformat/options_table.h); enum values `AVFMT_AVOID_NEG_TS_*`); required for clean MP4/MOV writes when input PTS are negative (typical with `-ss` + `-copyts`). All three render into the AVDict that [pipeline/muxer_options.go](pipeline/muxer_options.go)::`buildMuxerOptions` hands to `avformat_write_header` — `MuxDelay`/`MuxPreload` are seconds-to-microsecond-converted to the `max_delay`/`preload` AVDict integer keys, `AvoidNegativeTS` passes through as the canonical enum string libavformat parses. **`Config.StartAtZero bool`** modulates `Config.CopyTS`: under `-copyts` the runtime normally suppresses the demuxer-side ts_offset shift that rebases the first input PTS to 0 (preserving wall-clock timestamps for broadcast / HLS use); `StartAtZero=true` re-enables that shift even under `CopyTS` so the first kept packet still anchors at PTS 0 while later packet timing is left untouched. Mirrors FFmpeg's global `-start_at_zero` flag (the `start_at_zero ? 0 : f->start_time_effective` branch at [`fftools/ffmpeg_demux.c` L486](../ffmpeg/fftools/ffmpeg_demux.c)). Wired into [pipeline/handlers.go](pipeline/handlers.go) by adjusting the `tsOffsetUS` gate from `!CopyTS` to `(!CopyTS || StartAtZero)`. Validation: `MuxDelay`/`MuxPreload >= 0`; `AvoidNegativeTS` rejected against the enum at validate time so typos surface at config-load rather than only when libavformat rejects the AVDict; `StartAtZero` requires `CopyTS` (otherwise it would silently no-op — bug magnet). `compat/ffcli` parses `-muxdelay SECONDS` / `-muxpreload SECONDS` / `-avoid_negative_ts MODE` (latched onto the next output) and `-start_at_zero` (global, latched onto the Config). Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `muxdelay` / `muxpreload` / `avoid_negative_ts` with the enum + non-negative numeric constraints; top-level `start_at_zero` boolean); mirrored in `frontend/src/lib/jobTypes.ts`. Regression coverage: [pipeline/muxer_options_test.go](pipeline/muxer_options_test.go) (`TestBuildMuxerOptionsTimestampPolicy` proves the seconds → microseconds conversion lands on the right AVDict keys; `TestValidateStartAtZeroRequiresCopyTS` and `TestValidateAvoidNegativeTSEnum` cover the new validators) and [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go) (`TestParseTimestampPolicy` round-trips all four flags in one command line; `TestParseTimestampPolicyErrors` covers negative `muxdelay`/`muxpreload` + bogus `avoid_negative_ts`). Out of scope for this PR: the per-output `-output_ts_offset` AVOption (continues to work via `Options`), the negotiation between `-copyts` + `-start_at_zero` + per-input `-itsoffset` (already composes additively today via [pipeline/handlers.go](pipeline/handlers.go) — Wave 5 #28 covered `Input.ITSOffset`), and the deprecated `-vsync drop` alias for `avoid_negative_ts=disabled` (use the explicit field).

- **`setsar` / `setdar` shorthand on `Output` (`Output.SAR` + `Output.DAR` / FFmpeg `-aspect A:B` + `setsar=A:B` / `setdar=A:B`).** Wave 3 #15 of §6 of the parity plan ([docs/ffmpeg-coverage-roadmap.md](docs/ffmpeg-coverage-roadmap.md)). Two optional string fields land the universally-requested per-output sample / display aspect-ratio overrides that the legacy SD corpus depends on (DV-PAL 720×576 @ 4:3 → SAR 16:15; NTSC 720×480 @ 4:3 → SAR 8:9; HD square pixels 1920×1080 @ 16:9 → SAR 1:1) without forcing the user to insert a `setsar`/`setdar` filter node by hand. Both fields accept the canonical FFmpeg grammar (`A:B`, `A/B`, or decimal float, parsed by [pipeline/aspect.go](pipeline/aspect.go)::`parseAspectRatio` mirroring `av_parse_ratio`); `SAR` is written verbatim onto the encoder's `AVCodecContext.sample_aspect_ratio` (which `avcodec_parameters_from_context` then propagates to `AVStream.codecpar.sample_aspect_ratio`); `DAR` is resolved at encoder-open time against the encoder's just-decided width/height (`resolveSAR`: SAR_num/den = (DAR_num × H) / (DAR_den × W), reduced via Euclidean gcd) — so the same `dar: "4:3"` line yields the right SAR regardless of whether the upstream filter chain produced 720×576, 720×480, or 1920×1080 frames. `SAR` and `DAR` are mutually exclusive at validate time (rejected with a clear "sar and dar are mutually exclusive" error). New av-layer surface: `av.EncoderOptions.SampleAspectRatio [2]int` is forwarded to `AVCodecContext.sample_aspect_ratio` in [av/encode.go](av/encode.go)::`OpenEncoder` between the framerate / pix_fmt / GOP block and `avcodec_open2`, so the muxer sees the configured SAR before `WriteHeader`. Wired through `expandImplicitEncoders` ([pipeline/handlers.go](pipeline/handlers.go)) which propagates `Output.SAR` / `Output.DAR` onto the synthetic encoder node as `__sar` / `__dar` (only on video edges); `createEncoder` resolves the pair after `Width`/`Height` are decided so DAR can convert against the actual encoder dimensions. `compat/ffcli` parses `-aspect VALUE` (the legacy form, rewritten to `Output.DAR` per §6.8 of the roadmap), plus `-setsar` / `-setdar` as explicit aliases for the matching field. Schema fields added to both `schema/v1.0.json` and `schema/v1.1.json` (output-level `sar: string` + `dar: string`); mirrored in `frontend/src/lib/jobTypes.ts`. Capability registry bumped (`setsar / setdar`: partial → covered with schema pointer at `pipeline.Output.SAR / pipeline.Output.DAR`; `-aspect`: missing → covered at `pipeline.Output.DAR` with the importer-rewrites note). Roadmap §6.3 #15 marked landed. Regression coverage: [pipeline/aspect_test.go](pipeline/aspect_test.go) covers `parseAspectRatio` (6 valid forms — `A:B` / `A/B` / decimal / whitespace-tolerant), the 8 invalid-input cases (`""`, `"abc"`, `"16:"`, `":9"`, `"16:0"`, `"0:9"`, `"-1:1"`, `"16:9:1"`), the validator (SAR-only / DAR-only / mutually-exclusive / bad-SAR / bad-DAR paths), and the SAR resolver (canonical SAR passthrough plus DAR → SAR for DV-PAL 720×576 / NTSC 720×480 / HD 1920×1080 plus the empty-input no-op + zero-dimensions error). ### Fixed
- **`AVFMT_NOFILE` muxers (`hls`, `dash`, `segment`, `tee`) no longer race the wrapper's atomic `.tmp → final` rename.** `av.OpenOutputWithFormat` previously stamped `tmpPath = path + ".tmp"` and `outPath = path` on every returned `OutputFormatContext`, so `Close()` always tried `os.Rename(path+".tmp", path)`. For file-based muxers that worked because we had `avio_open`'d the `.tmp` file ourselves; for `AVFMT_NOFILE` muxers (which manage their own files via `AVFormatContext.url`, and which — for hlsenc/dashenc — implement their *own* atomic `.tmp` rename internally controlled by their `temp_file` AVOption) it produced a guaranteed `rename …: no such file or directory` failure on close because the muxer had already moved its own `.tmp` to the final name. `OpenOutputWithFormat` now returns an `OutputFormatContext` with empty `tmpPath`/`outPath` for the `AVFMT_NOFILE` branch so `Close()` skips the wrapper rename entirely. Surfaced by the new `41_hls_vod.json` and `42_dash_basic.json` end-to-end smokes.
- **Positional FFmpeg-style filter args (`scale=320:240`) reach libavfilter intact.** `compat/ffcli/ffcli.go.parseFilterExpr` records each `:`-separated token without an embedded `=` as `_pos0`, `_pos1`, ... so the original order can be recovered, but `pipeline/engine.go.buildFilterSpec` previously emitted those keys verbatim, producing `scale=_pos0=320:_pos1=240` which `avfilter_graph_parse_ptr` rejected with `Could not set non-existent option '_pos0'`. The builder now partitions params into positional (`_posN`) and named, sorts positional by index, and emits them as bare in-order values before any named args. Surfaced by the `vf_scale_positional_x264_audio_copy` round-trip case.
- **libx264 (and any encoder) fed by a filter graph no longer blows up the container duration.** When the upstream filter graph's buffersink advertised a finer `time_base` than `1/framerate` (e.g. demuxer TB `1/12288` propagated through `scale`), the encoder was opened with `time_base = 1/framerate` only, so frame PTS produced in the buffersink's units were reinterpreted in the encoder's coarser units — a 24 fps clip ran out at ~512× the real duration (219 frames → ~4650 s instead of ~9.1 s). `av.EncoderOptions` gained a `TimeBase [2]int` field that, when set, overrides the framerate-derived default; `pipeline/handlers.go.createEncoder` now populates it from `fg.OutputTimeBase(0)` whenever the encoder's upstream is a `FilterGraph`. Surfaced by the `vf_scale_named_x264_no_audio` round-trip case.

- **Runtime: input-side `-ss` / `-t` / `-to` honoured by the demuxer** as a faithful Go port of `fftools/ffmpeg_demux.c`. `-ss` triggers an `avformat_seek_file(ctx, -1, INT64_MIN, target, target, 0)` (with the container's reported `start_time` added when known); `-t` / `-to` resolve through the same conflict rules ffmpeg.c uses (mutually exclusive — `-t` wins with a warning; `-to` alone converts to `recording_time = to - max(ss, 0)`; `-to <= -ss` is rejected); the demux loop then enforces `recording_time` per packet, mirroring the `dts >= recording_time + start_time` check in `input_packet_process()`. Every emitted packet's PTS/DTS is also shifted by `ts_offset = -timestamp` to mirror `ts_fixup()` in non-`copy_ts` mode, so a `-ss 450 -t 10` stream-copy produces a 10-second file (PTS [0, 10s]) instead of a 460-second container with a 10-second hole at the front. Time strings are parsed by binding `av_parse_time(..., 1)` exactly as the FFmpeg CLI does, so any value the FFmpeg command line accepts (`30`, `5.5`, `00:01:23.250`, `90s`) Just Works. Implemented in [pipeline/timing.go](pipeline/timing.go), with the seek + per-packet stop check + ts_offset shift wired into [pipeline/handlers.go](pipeline/handlers.go); new `av.ParseTime` / `av.InputFormatContext.StartTime` / `av.NoPTSValue` / `av.Packet.ShiftTS` / `av.Packet.SetPTS` / `av.Packet.SetDTS` exposed in [av/demux.go](av/demux.go) and [av/packet.go](av/packet.go). Regression coverage in [pipeline/timing_test.go](pipeline/timing_test.go).
- **Stream-copy nodes (`type: "copy"`).** A new node category lets a graph forward demuxer `AVPacket`s straight to the muxer with no decode/encode — the standard "swap container, keep streams" pattern. One copy node per media type (`copy_video`, `copy_audio`, `copy_subtitle`, `copy_data`) is exposed under the new **Copy** palette section between Encoders and Processors. The implicit-encoder splice treats copy nodes as already-encoded, so a manual `in:a:0 → copy_audio → out:a` edge no longer gets an AAC encoder spliced in. The runtime adds the output stream via `avcodec_parameters_copy` (no encoder context required), rescales packet timestamps from the demuxer's `time_base` to the muxer's post-header `time_base`, and clones packets when one demuxed stream feeds multiple copy nodes. Implemented across `graph/graph.go` (new `KindCopy`), `av/packet.go` (`Packet.Rescale`, `ClonePacket`), `av/mux.go` (`AddStreamFromInput`, `StreamTimeBase`), `pipeline/handlers.go` (new `handleCopy` + per-stream-type packet routing in `handleSource` + per-channel rescale in `handleSink`), `internal/gui/api.go` (palette entries + new `"Copy"` category in `categoryOrder`), and `frontend/src/lib/spawn.ts` / `MMNode.tsx` / `jsonAdapter.ts`. Documented in [docs/json-config-reference.md](docs/json-config-reference.md) and [docs/gui.md](docs/gui.md).
- **ABR ladder example** (`testdata/examples/35_abr_ladder.json`) — single input fans out to four `scale` filters (1920×1080, 1280×720, 960×540, 640×360), each feeds a dedicated `libx264` encoder at 7000/4000/2000/1000 kbps, with a single AAC audio encoder fanned out to all four MP4 muxers. Exercises explicit fan-out, multi-output mux, and per-encoder bitrate independence end-to-end.

### Fixed
- **GUI: Imported stream-copy nodes now match user-spawned ones.** Materialised copy nodes (synthesised by `materializeImplicitEncoders` from a `codec: "copy"` output) used `auto_enc_<output>_<type>` ids and rendered with all four media-type pins, so a `ffmpeg -c copy` import looked nothing like the `Copy (video)` / `Copy (audio)` cards from the palette. Visual parity with palette-spawned copy nodes restored.
- **GUI Import FFmpeg dialog: stream-copy commands no longer crash.** `POST /api/convert-cmd` produced a config whose `graph.nodes` was the JSON literal `null` (Go's nil-slice marshalling) for any command without filters or explicit encoders — e.g. `ffmpeg -i in -c copy -t 30 out`. The frontend then crashed in `materializeImplicitEncoders` calling `.map()` on null. The `compat/ffcli` parser now always emits non-nil empty `Nodes` / `Edges` slices, and `materializeImplicitEncoders` defensively coerces null array fields. Regression tests added in [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go).
- **`compat/ffcli`: `-t`, `-ss`, `-to` are now preserved.** Previously these timing flags were consumed by the catch-all `-flag value` branch and dumped into an unused `globalOpts` bucket, silently dropping them. They are now attached as `Options` entries on the next file specifier on the command line (input or output), matching FFmpeg's positional semantics — so `ffmpeg -i in -c copy -t 30 out` round-trips with `outputs[0].options.t = "30"` and the demuxer/muxer pass-through honours it.
- **GUI: CRF / QP defaults no longer show as `-1`.** libavcodec encodes "unset, pick the codec's built-in default" as the sentinel `(default=-1, min=-1, max=FLT_MAX)` for CRF/QP-style options on libx264/libx265/libsvtav1/libvpx_vp9/libaom_av1. The Inspector now substitutes a per-codec override table (`ENCODER_PRIMARY_OVERRIDES`) so the placeholder reads `(default — 23)` and the slider/number bounds are realistic (e.g. libx265 CRF 0–51 with default 28).
- **GUI: edges serialise using the node id, not the display label.** `deriveEndpoint` was returning `node.data.label` for graph nodes, which is the codec/filter display name (e.g. `aac`, `libx264`); a hand-wired `enc_audio` was being saved as `to: "aac"` and the runtime then failed with `unknown destination node aac`. The graph-node branch now emits the React Flow id (which equals the job-side id).
