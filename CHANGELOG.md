### Added
- **Migrated `__*` sentinel `Params` keys to typed
  `graph.NodeDef.Internal` (Milestone B.4–B.6).** The lowering
  helpers in [pipeline/handlers_graph_build.go](pipeline/handlers_graph_build.go)
  and [pipeline/loudnorm.go](pipeline/loudnorm.go) now write
  `Internal.Encoder` (FPSMode, ForceKeyFrames, SAR, DAR,
  EncoderTimeBase, FieldOrder, Interlaced, Pass, PassLogFile,
  PassIndex), `Internal.Filter` (Threads, LoudnormPass,
  LoudnormStatsFile), and `Internal.Generated` (provenance for
  synthesised nodes — implicit encoders, audio sample-fmt /
  audio_sync resamplers, two-pass loudnorm shuttle). The encoder
  and filter consumers
  ([pipeline/handlers_encoder.go](pipeline/handlers_encoder.go),
  [pipeline/handlers_filter.go](pipeline/handlers_filter.go)) read
  from those typed fields. `encoderReservedParams` is trimmed to
  the runtime-special-cased non-AVOption keys
  (`codec, width, height, bitrate, threads, thread_type`); all
  `__*` entries are gone. `buildFilterSpec` keeps its `__` prefix
  strip as a defensive guard against hand-edited authoring
  configs.
- **Typed `graph.NodeDef.Internal` lowering bag.** Adds an
  `Internal` field to `graph.NodeDef` and `graph.Node` carrying
  per-kind sub-structs (`EncoderInternal`, `FilterInternal`) plus a
  `Generated {By, From, Reason}` provenance record for synthesised
  nodes. The struct is the typed replacement for the historical
  `__*` sentinel keys in `Params`; this commit lands the types and
  plumbs them through `graph.Build` but does not yet write or read
  them — that follows in Milestone B.4/B.5/B.6. See
  [graph/internal_fields.go](graph/internal_fields.go) and
  [docs/field-ownership.md](docs/field-ownership.md).
- **`pipeline.NormalizeConfig` boundary.** New
  `pipeline.NormalizeConfig(cfg) (*graph.Def, []NormalizeWarning, error)`
  is the single, deterministic entry point for lowering an authoring
  `pipeline.Config` into the executable `graph.Def` the runtime
  consumes. `runGraph` now goes through it and surfaces any returned
  warnings as `ErrorEvent`s on the existing pipeline events channel.
  Today the function is a thin wrapper around `configToGraphDef`;
  subsequent commits move the `__*` sentinel-key lowering behind it
  and replace the sentinels with a typed `NodeDef.Internal` sub-struct
  (Milestone B of `private_local/normalization_plan_revised.md`).
  Three regression tests in
  [pipeline/normalize_test.go](pipeline/normalize_test.go) gate the
  no-mutation, determinism, and shorthand-coverage contracts.
- **Field ownership classification.** New
  [docs/field-ownership.md](docs/field-ownership.md) classifies every
  `Config` / `GlobalOptions` / `Output` / `Input` field as node-local,
  authoring shorthand, muxer-owned, true global, or deferred.
  Authoritative reference for the normalization-boundary work tracked
  in `private_local/normalization_plan_revised.md` (Milestone A).
  No code changes; documentation only.

 Closes the usability gap reported against the per-filter Inspector (the three top fields `ID` / `Type` / `Filter` were always-editable text inputs that almost never wanted to be changed, every option row showed `int · default 0 · ≤ 1` then `set mode` as separate dim lines, and booleans were rendered as a `(default) / true / false` `<select>` even when the underlying AVOption was an `int` with `min=0 max=1`). Design captured in [docs/proposals/filter-properties-redesign.md](docs/proposals/filter-properties-redesign.md). Implemented:
  - **Header refactor** — for `filter`, `filter_source`, and `filter_sink` graph nodes the three structural identifiers (`id`, `type`, `filter`) collapse into a single `▸ Advanced` disclosure in [frontend/src/components/Inspector.tsx](frontend/src/components/Inspector.tsx) (`FilterAdvanced`). Inside the disclosure the type is read-only (it is determined when the node is created from the Palette) and the filter swap goes through a `Replace…` confirmation that warns the user it discards every option on the node. The Inspector heading + `describeKind` subtitle ("Audio filter" / "Video filter") already conveyed the kind, so the body of the panel now leads with the filter's own description and option list.
  - **Option-row layout** — the new `.filter-option-row` block in [frontend/src/styles.css](frontend/src/styles.css) reorders each AVOption to: `NAME` (uppercase label) → help text in body colour as the row's primary descriptive line → control on the left of a flex row → quiet `0.015625–64.0` range badge tucked against the right edge. The `default <n>` token is gone (the input's placeholder shows it instead) and so is the `<type>` token (the range formatting itself conveys the type — see below). The badge is suppressed entirely for enums, booleans, strings, and options whose min/max are both libavutil sentinels.
  - **Type-aware range formatting** — `rangeHint` in [frontend/src/components/FilterForm.tsx](frontend/src/components/FilterForm.tsx) renders `float` / `double` / `rational` endpoints with at least one decimal place (`0–1` becomes `0.0–1.0`, `0.015625–64` becomes `0.015625–64.0`) and integer-typed endpoints as plain integers (no spurious `.0`). This makes the range visually disambiguate the type without needing a separate token.
  - **Booleans never render as numbers.** `BoolToggle` in [frontend/src/components/controls/OptionControl.tsx](frontend/src/components/controls/OptionControl.tsx) replaces the old `(default) / true / false` `<select>` with a segmented `true` / `false` toggle plus a trailing `✕ clear` that drops the override. The toggle is also used for "implicit booleans" — `int`-typed AVOptions with `min=0 max=1` and named consts like `on=1` / `off=0`, `yes=1` / `no=0`, `enabled=1` / `disabled=0` (e.g. `acompressor`'s `mode` field). The committed string is the word `true` / `false` (libavutil's `av_opt_set` accepts both words and digits, so it round-trips cleanly).
  - **Integer rounding on commit.** `NumericInput` in `OptionControl.tsx` now uses commit-on-blur semantics. Integer-typed options round fractional input to the nearest valid integer (`Math.round` then clamp to `[min, max]`); float / double options accept integers verbatim and leave fractional input untouched. A pasted `2.6` into an `int` field commits as `3`, not as a libavutil parse error downstream. Non-finite input reverts to the previously saved value.
  - **Modified-options indicator** — a small accent-coloured `●` floats to the left of the option name when `def.params` carries a non-empty value for that key. Surfaces "what did I override?" at a glance.
  - **Out of scope (deferred)** — `unit`-grouped option sections, "Reset to defaults" / generated-CLI footer, container-specific metadata key autocomplete; these remain open in the proposal doc.

### Fixed
- **GUI progress bar no longer jumps to 100% while a slow video encoder is still running.** Previously every output packet — regardless of stream — bumped a single per-node monotonic-max `outputPTSNs` atomic in [pipeline/metrics.go](pipeline/metrics.go), so when a multi-stream sink combined a fast encoder (e.g. AAC at ~300 fps) with a slow one (e.g. libx265 `slow` / `crf=16` at ~3 fps), the audio stream's packets reached the muxer's end-of-file in seconds and pinned the metric at 100% even though video encoding was minutes from finishing — the GUI status line would read `media: 09:56 / 09:56 (100.0%) · 6.38× realtime · ETA 00:00` while libx265 was at packet 305 with most of the file ahead. New `NodeMetrics.AdvanceOutputPTSStream(streamIdx, d)` keeps a per-stream monotonic-max in `streamOutputPTS map[int]int64` (guarded by `streamOutputMu sync.RWMutex`); `NodeMetrics.Snapshot()` reports `OutputPTS` as the *minimum* across streams that have written ≥ 1 packet (the slowest stream is the one that determines when the run actually finishes), falling back to the legacy aggregate when no per-stream entry exists yet. The existing cross-sink min aggregation in `MetricsRegistry.Snapshot()` composes correctly on top: a multi-output graph reports min-of-min. [pipeline/handlers_sink.go](pipeline/handlers_sink.go)::`writeOne` now calls `AdvanceOutputPTSStream(i, ptsNs)` with the stream index already in scope. Regression coverage: [pipeline/metrics_test.go](pipeline/metrics_test.go) `TestNodeMetrics_OutputPTSPerStreamMin` (writes PTS 1 s on stream 0 and 10 min on stream 1, asserts Snapshot returns 1 s; advances video to 5 s, asserts 5 s; out-of-order rewind on a stream is ignored) and `TestNodeMetrics_OutputPTSLegacyFallback` (legacy `AdvanceOutputPTS` path still monotonic-max).
- **`AVFMT_NOFILE` muxers (`hls`, `dash`, `segment`, `tee`) no longer race the wrapper's atomic `.tmp → final` rename.** `av.OpenOutputWithFormat` previously stamped `tmpPath = path + ".tmp"` and `outPath = path` on every returned `OutputFormatContext`, so `Close()` always tried `os.Rename(path+".tmp", path)`. For file-based muxers that worked because we had `avio_open`'d the `.tmp` file ourselves; for `AVFMT_NOFILE` muxers (which manage their own files via `AVFormatContext.url`, and which — for hlsenc/dashenc — implement their *own* atomic `.tmp` rename internally controlled by their `temp_file` AVOption) it produced a guaranteed `rename …: no such file or directory` failure on close because the muxer had already moved its own `.tmp` to the final name. `OpenOutputWithFormat` now returns an `OutputFormatContext` with empty `tmpPath`/`outPath` for the `AVFMT_NOFILE` branch so `Close()` skips the wrapper rename entirely. Surfaced by the new `41_hls_vod.json` and `42_dash_basic.json` end-to-end smokes.
- **Positional FFmpeg-style filter args (`scale=320:240`) reach libavfilter intact.** `compat/ffcli/ffcli.go.parseFilterExpr` records each `:`-separated token without an embedded `=` as `_pos0`, `_pos1`, ... so the original order can be recovered, but `pipeline/engine.go.buildFilterSpec` previously emitted those keys verbatim, producing `scale=_pos0=320:_pos1=240` which `avfilter_graph_parse_ptr` rejected with `Could not set non-existent option '_pos0'`. The builder now partitions params into positional (`_posN`) and named, sorts positional by index, and emits them as bare in-order values before any named args. Surfaced by the `vf_scale_positional_x264_audio_copy` round-trip case.
- **libx264 (and any encoder) fed by a filter graph no longer blows up the container duration.** When the upstream filter graph's buffersink advertised a finer `time_base` than `1/framerate` (e.g. demuxer TB `1/12288` propagated through `scale`), the encoder was opened with `time_base = 1/framerate` only, so frame PTS produced in the buffersink's units were reinterpreted in the encoder's coarser units — a 24 fps clip ran out at ~512× the real duration (219 frames → ~4650 s instead of ~9.1 s). `av.EncoderOptions` gained a `TimeBase [2]int` field that, when set, overrides the framerate-derived default; `pipeline/handlers.go.createEncoder` now populates it from `fg.OutputTimeBase(0)` whenever the encoder's upstream is a `FilterGraph`. Surfaced by the `vf_scale_named_x264_no_audio` round-trip case.

Resolution counts streams of the requested media type in the order they were added to the muxer — the same convention FFmpeg's `cmdutils.c::check_stream_specifier` uses for `s:<type>:<idx>` — so `{Type:"a", Index:1}` resolves to the second audio AVStream regardless of how video / audio edges were declared on the sink. Wiring lives in `pipeline/handlers.go::handleSink`: a per-media-type counter is built up inside the AddStream loop, then a second pass walks `out.Streams` and applies `SetStreamMetadata` / `SetStreamDisposition` against the resolved stream index before `WriteHeader` (mirrors `of_add_metadata` running after stream creation in `fftools/ffmpeg_mux_init.c`). - **Per-output `-shortest` / `-fs` and output-side `-ss`/`-t`/`-to` with `-copyts` (`Output.Shortest` / `Output.MaxFileSize` / `Config.CopyTS`).** Three new fields land the canonical "stop at the shortest stream", "cap output at N bytes", and "trim at the muxer with optional absolute-PTS preservation" pattern that shows up in essentially every overlay / music-video / silent-video-plus-audio job, every capture-with-quota pipeline, and every broadcast-PTS workflow. Implementation mirrors the FFmpeg 8.0 reference end-to-end: `Output.Shortest` is enforced per-output (mirrors the per-`OutputFile` sync queue set up in `fftools/ffmpeg_mux_init.c`) by recording the PTS at which the first feeder channel of an output closes and then draining-and-dropping further packets on the remaining channels of the same output; `Output.MaxFileSize` mirrors `mux->limit_filesize` from `fftools/ffmpeg_mux.c::mux_packet_filter` by querying `avio_tell` on the muxer's IO context before every `WritePacket` and stopping with a clean trailer once the limit is reached (new `av.OutputFormatContext.BytesWritten` wraps the C call); output-side `-ss`/`-t`/`-to` from `Output.Options` resolve through the new `pipeline.resolveOutputTiming` (parallel to the input-side helper) and are enforced in `pipeline/handlers.go::handleSink` by dropping packets below `start_time` and stopping muxing at `start_time + recording_time`, mirroring `fftools/ffmpeg_mux.c::of_streamcopy` and `fftools/ffmpeg_enc.c::check_recording_time`. `Config.CopyTS` is a global flag mirroring `-copyts`: when true the demuxer-side `ts_offset` shift normally applied after `-ss` (see `fftools/ffmpeg_demux.c`) is suppressed so the original timestamps are preserved end-to-end, and the meaning of any output-side `-ss`/`-to` flips from "offset from input start" to "absolute timeline values" (`stopTimestampUS(true)` anchors at `startUS + recordingUS` rather than at `recordingUS`). Implemented across `graph/graph.go` (new `KindCopy`), `av/packet.go` (`Packet.Rescale`, `ClonePacket`), `av/mux.go` (`AddStreamFromInput`, `StreamTimeBase`), `pipeline/handlers.go` (new `handleCopy` + per-stream-type packet routing in `handleSource` + per-channel rescale in `handleSink`), `internal/gui/api.go` (palette entries + new `"Copy"` category in `categoryOrder`), and `frontend/src/lib/spawn.ts` / `MMNode.tsx` / `jsonAdapter.ts`. Documented in [docs/json-config-reference.md](docs/json-config-reference.md) and [docs/gui.md](docs/gui.md).
- **ABR ladder example** (`testdata/examples/35_abr_ladder.json`) — single input fans out to four `scale` filters (1920×1080, 1280×720, 960×540, 640×360), each feeds a dedicated `libx264` encoder at 7000/4000/2000/1000 kbps, with a single AAC audio encoder fanned out to all four MP4 muxers. Exercises explicit fan-out, multi-output mux, and per-encoder bitrate independence end-to-end.

### Fixed
- **GUI: Imported stream-copy nodes now match user-spawned ones.** Materialised copy nodes (synthesised by `materializeImplicitEncoders` from a `codec: "copy"` output) used `auto_enc_<output>_<type>` ids and rendered with all four media-type pins, so a `ffmpeg -c copy` import looked nothing like the `Copy (video)` / `Copy (audio)` cards from the palette. - **GUI Import FFmpeg dialog: stream-copy commands no longer crash.** `POST /api/convert-cmd` produced a config whose `graph.nodes` was the JSON literal `null` (Go's nil-slice marshalling) for any command without filters or explicit encoders — e.g. `ffmpeg -i in -c copy -t 30 out`. The frontend then crashed in `materializeImplicitEncoders` calling `.map()` on null. The `compat/ffcli` parser now always emits non-nil empty `Nodes` / `Edges` slices, and `materializeImplicitEncoders` defensively coerces null array fields. Regression tests added in [compat/ffcli/ffcli_test.go](compat/ffcli/ffcli_test.go).
- **`compat/ffcli`: `-t`, `-ss`, `-to` are now preserved.** Previously these timing flags were consumed by the catch-all `-flag value` branch and dumped into an unused `globalOpts` bucket, silently dropping them. They are now attached as `Options` entries on the next file specifier on the command line (input or output), matching FFmpeg's positional semantics — so `ffmpeg -i in -c copy -t 30 out` round-trips with `outputs[0].options.t = "30"` and the demuxer/muxer pass-through honours it.
- **GUI: CRF / QP defaults no longer show as `-1`.** libavcodec encodes "unset, pick the codec's built-in default" as the sentinel `(default=-1, min=-1, max=FLT_MAX)` for CRF/QP-style options on libx264/libx265/libsvtav1/libvpx_vp9/libaom_av1. The Inspector now substitutes a per-codec override table (`ENCODER_PRIMARY_OVERRIDES`) so the placeholder reads `(default — 23)` and the slider/number bounds are realistic (e.g. libx265 CRF 0–51 with default 28).
- **GUI: edges serialise using the node id, not the display label.** `deriveEndpoint` was returning `node.data.label` for graph nodes, which is the codec/filter display name (e.g. `aac`, `libx264`); a hand-wired `enc_audio` was being saved as `to: "aac"` and the runtime then failed with `unknown destination node aac`. The graph-node branch now emits the React Flow id (which equals the job-side id).
- **Build: `make build-debug` / `make build-gui-debug` — verbose build log for bug reports.** New Makefile targets capture a complete diagnostic log (`mediamolder-build.log`) that includes `go env`, gcc version, `pkg-config` cflags/libs for all six FFmpeg libraries, relevant environment variables (`PKG_CONFIG_PATH`, `CGO_CFLAGS`, `CGO_LDFLAGS`, `FFMPEG_SRC`), and the full `-v -x` Go build output (every compiler and linker invocation). `build-gui-debug` also captures the `npm run build` output. The log path is overridable via `LOG=` and build tags via `BUILD_TAGS=`. `mediamolder-build.log` added to `.gitignore`.
- **Build: `make check-deps` — clear prerequisite error before CGO cascade.** New target (wired into `make build` and `make build-gui`) checks for `gcc` and `pkg-config`-visible FFmpeg headers ≥ 8.1 (libavcodec ≥ 61) before starting any Go build.
